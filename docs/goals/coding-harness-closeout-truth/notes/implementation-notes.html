Tradeoffs

  • The state file now records the completed phased task ledger while goal.md remains the durable roadmap narrative and continuation prompt.
  • Existing dirty worktree changes are recorded as context, but not staged or modified by board creation.
  • Phase 0 did not add more generic standards. Existing standards work is treated as the baseline until the truth-layer implementation exposes a concrete gap.
  • The claim ledger is intentionally verifier-owned. Model summaries can point at claims, but report readiness is derived from structured evidence, not prose.
  • src/lib/pr-closeout.ts still exceeds the advisory ratchet target after Phase 1, so Phase 2 reduced it from 728 lines to 327 lines by moving blocker and status policy into focused modules.
  • src/lib/pr-closeout/claims.ts remains above the advisory file ratchet at 483 lines, but the blocking oversized function is gone. Further splitting waits for Phase 3 missing-context integration to provide a concrete reason.
  • Missing-context classes are a typed routing table and one deterministic classifier rather than prose or a broader policy engine. Replay and recovery can reuse the shape later if they prove that need.
  • Phase 4 uses the existing trace normalizer and the pre-existing redaction baseline already present in the worktree. It adds replay eval assertions without staging the unrelated trace-normalizer baseline edits.
  • The recovery contract denies unsafe handlers before any recovery hook can mutate state. That keeps recovery deterministic while the concrete handler registry remains unbuilt.
  • The first handler keeps rollback conservative: it removes the parent directory only when it is empty, otherwise records a stopped rollback skip instead of deleting content it did not create.
  • Browser rejected the local file:// notes URL under its URL policy, so the notes are served through localhost instead: http://127.0.0.1:8791/docs/goals/coding-harness-closeout-truth/notes/implementation-notes.html. The page now auto-refreshes every 10 seconds while the server is running.
  • The invariant gate uses source-text checks plus negative fixtures rather than AST parsing. That keeps Phase 7 dependency-free and reversible while still failing closed when required claim fields or false-success fixtures disappear.
  • Fixture presence is checked through stable test-title markers. This is less semantic than a full fixture manifest, but it avoids creating a second eval platform before Phase 8 decides whether skill or gate surfaces need compression.
  • Phase 8 treats JSC-283 packaged skill behavior assurance as the strongest current evidence. It proves local packaged skill behavior for the approved scope and explicitly defers release-gate promotion until reusable fixture runners exist.
  • pnpm observed:eval-usage was useful audit evidence but not a Phase 8 validation gate. It failed with the known tsx IPC EPERM issue, matching the existing command-truth gap in the alignment review. That should route to a command/tooling follow-up, not a packaged skill rewrite.

JSC-363 Browser Evidence Slice

Active slice: PU-042 / GAP-010, browser evidence as a routine gate. The implementation is intentionally placed in a deep module instead of adding browser-specific policy into the command body.

Layer Where Work Is Placed Why It Belongs There
CLI seam src/commands/evidence-verify.ts and src/lib/cli/registry/evidence-verify-command-spec.ts Accepts --browser-evidence and --browser-required-viewports, then delegates to the browser evidence module.
Deep module src/lib/browser-evidence/, including manifest-parser-items.ts and png-test-fixtures.ts Owns manifest parsing, item-level schema parity, screenshot proof, PNG nonblank checks, console policy, stable error codes, and the validation report.
Shared report contract src/lib/evidence/types.ts Adds browserEvidence to the existing evidence verification report without making browser evidence a delivery truth, review-state, or merge-readiness proof.
Docs and architecture ARCHITECTURE.md, README.md, docs/cli-reference.md, and src/lib/testing/README.md Keeps the human source map, CLI contract, and behavior-test ownership aligned with the new module.

Deep-module flow: CLI flags > evidence-verify command > browser-evidence validator > manifest parser / PNG inspection > evidence report. This preserves the thin public surface while making the enforcement path testable as a module.

Decisions made during implementation: manifests must include consoleEvents, consolePolicy, and blankScreenshotPolicy; malformed console policy fails closed; browser screenshot refs must be inspectable PNG artifacts; PNG files get deterministic pixel-level nonblank inspection. PR review follow-ups also tightened required viewport handling so CLI-supplied requirements are added to the manifest requirements instead of replacing or weakening them; parser/schema integer and unknown-property checks now stay aligned; malformed PNGs fail closed; and browser-evidence failures no longer downgrade higher-priority file error exit codes.
