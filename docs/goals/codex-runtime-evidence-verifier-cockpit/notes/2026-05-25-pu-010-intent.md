# PU-010 Implementation Intent

## Intent

Promote `delivery-truth/v1` from private fixture composition into a production
claim-verdict module that can be consumed by closeout code without broadening
`harness next --json` or adding a new public command surface. This slice should
keep local validation, external checks, review truth, tracker state, root
hygiene, and merge readiness separate at the verdict layer.

## Forbidden Change Set

- No new broad public `delivery-truth` command.
- No expansion of `harness next --json` into executable authority.
- No `src/commands/next*.ts` implementation in this slice. PU-010 is
  explicitly pr-closeout-library-integration only; PU-013 owns cockpit command
  metadata after production verdicts exist.
- No live GitHub, Linear, CodeRabbit, or CircleCI connector in this slice.
- No root-hygiene production validator; PU-011 owns root hygiene.
- No Codex runtime producer bridge; PU-012 owns producer/export behavior.
- No closeout claim may pass from local validation alone.
- No claim may pass without `evidenceRef`, `freshness`, `verifiedAt`, `source`, and `headSha` where the claim is PR-head-sensitive.

## Implementation Rules

1. Keep `src/lib/delivery-truth/index.ts` as a small facade over typed contract,
   composition, policy, and integration helpers.
2. Production verdicts must preserve the existing `pr-closeout/v1` vocabulary for
   status, freshness, blocker class, evidence refs, head SHA, and verification
   timestamp.
3. `merge_ready` must require separate current claim-support evidence from
   `external_state`, `review_state`, and `pr_closeout`.
4. Producer-provided TTL or freshness may orient the verifier, but verifier-owned
   policy must decide whether a claim is current enough to support a pass.
5. Stale, missing, unknown, mixed-head, non-claim-support, missing-artifact, and
   unresolved-review inputs must block or downgrade; they must not become pass
   verdicts.
6. Any `pr-closeout` integration must be additive and test-backed. Existing
   `pr-closeout/v1` behavior must remain available for callers that do not
   provide delivery-truth inputs.

## Rollback

Revert the production delivery-truth integration helpers and keep the private
fixture-level `composeDeliveryTruth` tests that prove false-success blockers.
If `pr-closeout` integration was added, restore the previous closeout behavior
while leaving negative delivery-truth fixtures in place.

## Acceptance Criteria

- Production `delivery-truth/v1` verdicts expose `claim`, `status`,
  `evidenceRef`, `evidenceRefs`, `source`, `headSha`,
  `verdictHeadSha`, `freshness`, `blockerClass`, `blockerCode`,
  `verifiedAt`, and `evidenceUse`.
- A valid `merge_ready` fixture passes only when separate current
  `external_state`, `review_state`, and `pr_closeout` claim-support receipts
  all match the verifier head SHA.
- Negative fixtures cover blended readiness, mixed-head evidence, producer TTL
  overreach, missing artifact receipt, stale external state, unresolved review
  state, and non-claim-support evidence.
- `pr-closeout` can consume or reference a production delivery-truth verdict
  without treating it as the only closeout truth source.
- No public command surface is added unless the intent review explicitly approves
  it as the narrowest integration path.
