# PU-009 Implementation Intent

## Intent

Implement the production packet contracts for \`review-state/v1\` and
\`external-state-snapshot/v1\` as narrow deep modules. The slice must promote the
PU-008 non-blending fixture model into typed, validated packet families without
wiring public closeout behavior or collapsing review truth, remote checks,
tracker state, local validation, and merge readiness into one status.

## Source Inputs

- Plan: \`.harness/plan/2026-05-24-codex-runtime-evidence-verifier-cockpit-plan.md\`, PU-009.
- Spec: \`.harness/specs/2026-05-24-codex-runtime-evidence-verifier-cockpit-spec.md\`, FR-008 / SA-008.
- Base branch: \`origin/codex/jsc-363-runtime-evidence-pu004\` after PR #298 merged PU-008 into the active stack base.
- External blocker note: PR #299 handles \`.harness\` version-control policy and is blocked by remote check refresh, so this slice must not depend on that PRs \`.harness/intent\` path.

## Allowed Change Set

- \`src/lib/review-state/**\`
- \`src/lib/external-state/**\`
- focused tests and fixtures for the new packet families
- \`docs/goals/codex-runtime-evidence-verifier-cockpit/**\` receipts or notes for this slice
- \`ARCHITECTURE.md\` only if the slice adds a durable public/deep module boundary that the architecture map must describe

## Forbidden Change Set

- No public delivery-truth command.
- No live GitHub, Linear, CodeRabbit, or CircleCI connector.
- No pr-closeout behavior change unless a test proves the integration point remains additive and non-blending.
- No broad \`.harness\` tracking policy change in this slice; PR #299 owns that correction.
- No merged readiness claim from a single summary status.

## Implementation Rules

1. \`review-state/v1\` must validate PR identity, head SHA, GitHub review state,
   CodeRabbit state, unresolved thread counts, reviewer artifact receipts, and
   validation ownership classifications.
2. \`external-state-snapshot/v1\` must validate source snapshots with \`fetchedAt\`,
   \`ttlSeconds\`, source status, \`headSha\` when PR-head-sensitive, stale flag,
   stale reasons, and evidence-use classification.
3. Stale, missing, unknown, or orientation-only external state may orient an
   agent, but must not be claim-supporting for delivery-ready or merge-ready
   semantics.
4. Reviewer artifact claims must remain receipt-backed; prose-only artifact
   claims must fail validation.
5. Packet modules must be small facades over typed validation helpers if the
   implementation grows beyond a single readable file.

Reviewer outputs must be non-empty artifacts under \`artifacts/reviews/\` and must

## Rollback

Remove \`src/lib/review-state/**\`, \`src/lib/external-state/**\`, and their tests.
If \`ARCHITECTURE.md\` was updated only for those modules, remove the matching
architecture note. Existing PU-001 through PU-008 foundation code remains valid.

## Acceptance Criteria

- \`review-state/v1\` accepts a valid unresolved-review fixture and rejects a
  missing reviewer-artifact receipt fixture.
- \`external-state-snapshot/v1\` accepts a current multi-source snapshot and
  rejects missing \`fetchedAt\`, missing PR-head-sensitive \`headSha\`, invalid TTL,
  stale head SHA, stale Linear snapshot, and CodeRabbit unavailable claim-support
  fixtures.
- Review-state tests include positive and negative coverage for every
  validation ownership classification enum used by reviewer artifacts or gate
  failures.
- Tests prove review-state refs and external-state refs remain separate packet
  families.
- No public closeout claim is added by this slice.
- Architecture documentation is updated if, and only if, the deep-module map
  gains a durable new boundary.
