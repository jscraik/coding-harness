## Scope
- This module covers Go code in downstream harness presets and any Go helpers added to this repository.
- Go standards apply when a project contains `go.mod`, `go.work`, `*.go`, or uses the `go-base` harness preset.

## Module and toolchain
- Go modules MUST keep `go.mod` and `go.sum` deterministic and committed together when dependencies change.
- Toolchain versions SHOULD be pinned through the repository runtime manager or documented CI image.
- Generated Go files MUST include their generator source or command contract.

## Style and APIs
- Go code MUST be formatted with `gofmt` or `go fmt ./...`.
- Public packages, exported identifiers, and command entrypoints SHOULD include useful doc comments.
- Keep package boundaries small and explicit; avoid large utility packages that hide domain ownership.
- Prefer standard library APIs unless an external dependency materially reduces risk or complexity.

## Errors and context
- Return errors instead of panicking across library and CLI boundaries.
- Wrap errors with actionable context while preserving `errors.Is` and `errors.As` behavior where applicable.
- Functions that perform I/O, network calls, subprocess execution, or long-running work SHOULD accept `context.Context`.
- Cancellation and timeout paths SHOULD be tested when they affect user-visible behavior.

## Testing
- Go behavior changes SHOULD include table-driven tests where the input matrix matters.
- Prefer real package tests over copied reproduction code.
- CLI behavior SHOULD be tested at the command boundary when argument parsing, environment handling, or exit codes change.

## Security
- Secrets MUST NOT be hard-coded in source, tests, fixtures, or generated examples.
- Validate and constrain filesystem paths, subprocess arguments, and network inputs before use.
- Dependency updates SHOULD run the repository security/audit contract plus Go-native vulnerability checks such as `govulncheck ./...` when available.
- Subprocess execution MUST use structured argument slices and validated executable paths; avoid shell construction for untrusted input.
- HTTP servers and clients SHOULD set explicit timeouts, body-size limits, and context cancellation on user-facing or network-facing paths.

## Enforcement
- Go changes MUST pass the repository baseline gate and Go-native checks where Go code is present:
  - `go fmt ./...`
  - `go test ./...`
  - `go vet ./...` when configured or required by the project
  - `govulncheck ./...` when dependencies, network/security surfaces, or Go module versions change and the tool is available
  - `pnpm check`
  - `bash scripts/validate-codestyle.sh`
  - `bash scripts/verify-work.sh --fast`
- Validation evidence MUST be explicit:
  - `Command: <exact command> -> pass|fail|blocked (<reason>)`
