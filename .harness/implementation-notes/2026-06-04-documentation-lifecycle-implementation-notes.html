Implementation Shape

This branch treats documentation as governed product surface. It adds lifecycle metadata, a manifest, a validator, docs-gate integration, PR-template evidence fields, and a controlled distinction between source docs, downstream-distributed docs, generated artifacts, and historical records.

Decisions Made During Implementation

Lifecycle metadata is required for governed docs, not every file in the repo. The plan called for lifecycle enforcement, but the useful boundary is governed documentation and promoted .harness artifacts. Local runtime output, caches, and advisory research stay outside hard enforcement unless explicitly promoted.
.harness artifacts are validated by role and state. Plans, specs, Linear handoffs, and audits can become execution inputs or durable evidence. The checker classifies them separately so Project Brain stays useful without making every temporary artifact canonical.
The docs lifecycle checker was split into deeper modules. The first implementation concentrated too much logic in src/lib/docs-surface/doc-lifecycle.ts. It now delegates frontmatter, path classification, distribution checks, .harness artifact checks, and reporting so the size ratchet stays green and future rules have a clearer home.
The distribution guard is source-to-template aware. Downstream projects should receive only intentionally distributed docs and templates. The guard checks source metadata against src/templates/** so Coding Harness doctrine can live in this repo without polluting customer projects by accident.
Reader-task evaluation is specified before implementation. The next issue should prove whether humans and agents can answer real lifecycle questions from the docs. The spec keeps that evaluation deterministic and advisory first, rather than immediately turning it into a merge-blocking LLM judge.

Tradeoffs

  • Strict enough to prevent drift, flexible enough to migrate. Legacy docs can be classified and brought under governance progressively; the branch does not try to rewrite the entire documentation estate in one pass.
  • SemVer is advisory for now. Metadata records downstream impact, but release automation is not yet blocked on every SemVer classification. That keeps the first rollout usable while follow-up issues harden policy.
  • Brand language is introduced as documentation, not a package rename. The repo can describe synAIpse as the AI Delivery Harness while preserving Coding Harness command/package compatibility until a separate migration is planned.

Things Changed While Proving

  • Biome reformatted several touched TypeScript files. Remaining unsafe style suggestions were repaired manually by converting string concatenation to template literals.
  • The initial lifecycle implementation exceeded the local file-size ratchet; that became the prompt to split the checker into smaller modules instead of waiving the warning.
  • The PR closeout template now has explicit documentation lifecycle and SemVer fields, so docs changes cannot hide inside a generic validation summary.
