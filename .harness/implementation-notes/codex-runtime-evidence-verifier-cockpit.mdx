Implementation decision: the validator stays dependency-free for now. It checks manifest shape, schemaVersion const alignment, example schemaVersion alignment, required \`schemaVersion\`, not-yet-emitted blocker metadata, structural example-vs-schema parity for the JSON Schema subset used by these contracts, existing TypeScript validator parity where validators exist, and negative drift fixtures. Reviewer-driven hardening: the first implementation review found that the validator was too shallow because it primarily checked \`schemaVersion\` and selected manifest invariants. The slice now validates nested required fields, enum/const/type constraints, arrays, date-time strings, patterns, sibling \`$ref\`, \`anyOf\`, and \`additionalProperties: false\`. It also fails if a schema introduces unsupported JSON Schema keywords such as \`oneOf\` without extending the validator, including when the unsupported keyword appears inside a referenced schema. That turns future schema-vocabulary drift into a deterministic failure instead of a remembered caution.
