## Actual Moat

- Proven: the moat is operational learning encoded into portable executable
  contracts, decision packets, fixtures, gates, and workflow proof.
- Strategic assumption: the moat compounds through repeated real PR failures
  converted into reliable repo behavior.
- Operating principle: complexity is not defensibility.

## Protect

- `harness next --json` as agent routing cockpit.
- `HarnessDecision` and structured gate outputs.
- `harness.contract.json` as published aggregate contract.
- CI required-check ownership and parity.
- Review readiness tied to current-head evidence.
- Downstream skill behavior proof.
- Learned-failure promotion into repo-local, provenance-aware memory or fixtures.

## Do Not Protect

- Command count.
- Governance prose volume.
- Optional tool/plugin breadth.
- Placeholder memory.
- String-only skill validation as readiness proof.
- Oversized orchestrators as permanent architecture.
- Compatibility layers without owner and sunset condition.

## Proof Rules

- Moat claims need eval, telemetry, fixture, or gate evidence.
- Operational outcomes matter more than artifact count.
- A system is moat-critical only if it improves review trust, execution
  determinism, portability, cognition, or learned-failure conversion.
