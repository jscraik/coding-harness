## Prerequisites
- Node.js `>=24.0.0`
- `pnpm@10.33.0` for this source repo
- `NPM_TOKEN` for private package install checks
- `jq`, `rg`, `fd`, `git`, and `mise` when available
- GitHub token or GitHub App JWT only for remote checks

## Command Truth
Start fresh agents from:

```bash
harness next --json
```

First-contact help and catalogs:

```bash
harness --help
harness commands --json --for-agent
harness commands --json
harness --help --all-commands
```

When maintaining this source repo, avoid stale globals:

```bash
pnpm exec tsx src/cli.ts next --json
pnpm exec tsx src/cli.ts commands --json
```

## Install
Consumer repos use the installed binary:

```bash
mise install -g npm:@brainwav/coding-harness
harness --version
```

Fallback:

```bash
npm install -g @brainwav/coding-harness
```

CircleCI private package auth belongs in project environment variables:

```bash
# CircleCI project settings -> Environment Variables:
#   NPM_TOKEN
```

## Bootstrap
First-time scaffold:

```bash
harness init --dry-run
harness init
harness init --check-updates
```

Do not pass provider-specific init flags here; current init defaults and
contract state own the provider choice.

## Upgrade
Routine update:

```bash
harness init --check-updates
harness upgrade --dry-run
harness upgrade
```

Tracked baseline re-scaffold or contract transition:

```bash
harness init --update
harness init --migrate
harness init --rollback
```

Preview artifacts may include `--update --dry-run --json`, `updated`, `skipped`,
`created-to-updated compatibility alias`, `adoption-preview`, `tracked-update`,
and safe in-repo symlinked directories.

## CI Migration
Snapshot-backed CircleCI migration:

```bash
harness ci-migrate prepare --provider circleci --dry-run
harness ci-migrate prepare --provider circleci --apply
harness ci-migrate verify --snapshot <snapshot-id>
harness ci-migrate commit --snapshot <snapshot-id>
harness ci-migrate abort --snapshot <snapshot-id>
```

Use `abort` if verification fails or rollback is requested.

## Environment Sync
`harness init` may generate `.codex/environments/environment.toml` from canonical
Tools, Run, Debug, and Test action blocks plus current project scripts. Only
auto-update that file when it is harness-autogenerated or explicitly approved.

## Validation
Source repo:

```bash
pnpm skill:validate
pnpm check
pnpm test:deep
bash scripts/validate-codestyle.sh
```

Consumer repo:

```bash
bash scripts/codex-preflight.sh --stack auto --mode required
harness check-environment --json
harness docs-gate --mode advisory --json
harness verify-coderabbit
harness check-authz --contract harness.contract.json --repo <owner/repo> --branch <feature-branch>
```

Auth-bound checks are blocked or skipped when credentials are missing. Use a
real allowed repo and work branch for authorization checks; `main`, `master`,
and `release/*` are protected-branch negative tests, not green-path validation
targets.

The following workflow-contract section is intentionally kept in this reference
because `pnpm run workflow:validate` uses it as a compact command-lifecycle
fixture.

## Invariants
- Every transition row preserves canonical `S | E | G | A | N` semantics.
- Terminal states emit deterministic machine-readable outcomes: `DONE`, `FAIL`, or `BLOCKED`.

## States
- `PREFLIGHT`: confirm root, tools, auth posture, and command surface.
- `PREVIEW`: run dry-run or non-mutating discovery.
- `APPLY`: run selected scaffold, upgrade, or migration command.
- `VALIDATE`: run focused then broader gates.
- `DONE`, `FAIL`, `BLOCKED`: terminal outcomes.

## Transition Table (Canonical)
`S | E | G | A | N`

| S | E | G | A | N |
| --- | --- | --- | --- | --- |
| `PREFLIGHT` | `context_ok` | root, tools, and auth posture are known | run live command discovery | `PREVIEW` |
| `PREVIEW` | `preview_ok` | dry-run output is reviewed | run selected apply command | `APPLY` |
| `APPLY` | `apply_ok` | selected mutation exits cleanly | run validation ladder | `VALIDATE` |
| `VALIDATE` | `checks_pass` | blocking gates pass | report evidence and residual risk | `DONE` |
| `*` | `missing_dependency` | tool, auth, or repo prerequisite is absent | record blocker | `BLOCKED` |
| `*` | `error` | command or policy gate fails | stop and report failing command | `FAIL` |

## Error Handling
- `VALIDATION_ERROR`: invalid invocation, malformed data, or missing required input.
- `BLOCKED_DEPENDENCY`: missing binary, auth, network, or repo prerequisite.
- `POLICY_FAIL`: required validation, policy, or governance gate failed.
- `SYSTEM_ERROR`: unexpected runtime or process failure.

## Idempotency
- Discovery and preview commands must be side-effect free.
- Apply commands must be guarded by current repo state and safe to retry where the CLI supports replay.
- Snapshot-backed CI migration must preserve rollback data until `commit` succeeds.

## Execution Modes
- `STRICT`: stop on first blocking validation or policy failure.
- `ADVISORY`: collect warnings and continue only where the command contract allows it.

## Dry-Run Simulation
Dry-run commands have no side effects and must emit a deterministic trace of
planned writes, gates, or migration steps. Treat any mutation during dry-run as
`SYSTEM_ERROR`.

## Observability Logs
Record `workflow_id`, `transition_code`, `from_state`, `to_state`,
`correlation_id`, and `result` for workflow evidence.

## Validation Checklist
- Canonical `S | E | G | A | N` table exists.
- Every non-terminal state has an outbound path.
- Failure paths route to `FAIL` or `BLOCKED`.
- Dry-run behavior is side-effect free and deterministic.

## Boundaries
Harness can scaffold governed files, run local gates, emit JSON evidence,
manage snapshot-backed CI migration, and verify remote policy when credentials
exist.

Harness cannot create tokens, install GitHub Apps, bypass branch protection,
turn missing auth into a pass, or overwrite user-owned environment files without
approval.
