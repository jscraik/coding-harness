## Purpose
Use this guide when installing or repairing `@brainwav/coding-harness` in a
consumer repository. In the source repo, prefer `node --import tsx src/cli.ts ...`;
in consumer repos, use the installed `harness` binary.

## Preflight
```bash
pwd
git status --short --branch
harness next --json
harness --help
harness commands --json --for-agent
harness commands --json
```

If the repo owns a preflight wrapper:

```bash
bash scripts/codex-preflight.sh --stack auto --mode required
```

## Auth and Install
Do not print secret values. Check only presence or auth status.

```bash
test -n "${NPM_TOKEN:-}"
npm whoami
npm view @brainwav/coding-harness dist-tags --json
mise install -g npm:@brainwav/coding-harness
harness --version
```

Fallback:

```bash
npm install -g @brainwav/coding-harness
```

`NPM_TOKEN` belongs in local shell auth and CircleCI project environment
variables, not the standard GitHub Actions lane.

## Bootstrap
Preview first, then apply:

```bash
harness init --dry-run
harness init
harness init --check-updates
```

Do not pass CI-provider flags to `harness init`; current init defaults and
contract state own the provider choice.

## CI Migration
Use the snapshot-backed migration lane whenever GitHub Actions PR workflows must
move to CircleCI. Never delete `.github/workflows/` manually.

```bash
harness ci-migrate prepare --provider circleci --dry-run
harness ci-migrate prepare --provider circleci --apply
harness ci-migrate verify --snapshot <snapshot-id>
harness ci-migrate commit --snapshot <snapshot-id>
harness ci-migrate abort --snapshot <snapshot-id>
```

CircleCI owns the primary PR gate. CodeRabbit is the independent review check.
Semgrep Cloud is the independent external security check. GitHub Actions remains
release or fallback only unless an intentional migration changes the contract.

## Existing Repos
Routine upgrades use `harness upgrade`, not `harness init --update`:

```bash
harness init --check-updates
harness upgrade --dry-run
harness upgrade
```

Use re-scaffold commands only when tracked baseline files must be rewritten:

```bash
harness init --update
harness init --migrate
harness init --rollback
```

For upgrade-matrix evidence, `harness init --update --dry-run --json` may emit
`updated`, `skipped`, `created-to-updated compatibility alias`,
`adoption-preview`, `tracked-update`, and safe in-repo symlinked directories.

## Validation
```bash
harness check-environment --json
harness docs-gate --mode advisory --json
harness verify-coderabbit
harness check-authz --contract harness.contract.json --repo <owner/repo> --branch <feature-branch>
pnpm check
pnpm test:deep
```

Remote checks are auth-bound. Missing credentials means blocked or skipped, not
passed. Use a real allowed repo and work branch for authorization checks; `main`,
`master`, and `release/*` are protected-branch negative tests, not green-path
install validation targets.

## Owned Surfaces
Harness-managed or generated surfaces include `harness.contract.json`,
`.harness/ci-required-checks.json`, `memory.json`, `.circleci/config.yml`,
`.coderabbit.yaml`, `scripts/check-environment.sh`, and
`.codex/environments/environment.toml` when harness-autogenerated.

Project-owned starter surfaces include PR templates, CODEOWNERS,
`CONTRIBUTING.md`, `Makefile`, `biome.json`, `prek.toml`, `.mise.toml`, and
`scripts/codex-preflight.sh`.

## Boundaries
Harness can scaffold governance files, produce command/gate evidence, verify
local policy state, run auth-bound remote checks when credentials are provided,
and generate Codex environment action blocks from project scripts.

Harness cannot create credentials, install GitHub Apps, bypass branch
protection, make skipped auth-bound checks pass, or safely overwrite
user-customized `.codex/environments/environment.toml` without explicit
approval.
