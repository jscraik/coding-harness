## Findings (severity-ranked)

### 1) High - Replay corruption when runId is reused: old events are silently chained into a new run
- Severity: high
- Confidence: 96
- Evidence:
  - `src/commands/runtime-card.ts:161-164` always starts recording into whatever run directory `--trace-out` points to.
  - `src/lib/runtime-trace/runtime-card-trace.ts:213-217` appends events via `appendCanonicalEvent` with no run reset guard.
  - `src/lib/contract/run-records-core.ts:958-1005` explicitly appends to existing `events.jsonl` when present.
  - `src/lib/runtime-trace/runtime-card-trace.ts:329-359` overwrites the manifest for the same runId, but does not clear old events.
- Failure scenario:
  1. Operator runs `runtime-card --trace-out artifacts/agent-runs/R/events.jsonl` once; events and manifest are written.
  2. Operator reruns later with the same runId `R` (legitimate repetition abuse).
  3. New events are hash-chained after old events; manifest timestamps/outcome describe only the second invocation.
  4. Replay consumer loads `R` and gets a mixed multi-execution stream under one runId, violating single-run replay expectations and making forensic conclusions ambiguous.
- Impacted behavior:
  - Canonical replay stream is no longer one execution per runId.
  - Advisory evidence can mislead triage by blending runs.
- Remediation:
  - Fail closed when `events.jsonl` or `manifest.json` already exists for `runId`, unless an explicit `--trace-append` mode is introduced.
  - Add regression test: reusing `--trace-out` runId must error (or produce a fresh, isolated run namespace).

### 2) Medium - Concurrent writers to same runId can lose events (last-writer-wins rename race)
- Severity: medium
- Confidence: 82
- Evidence:
  - `src/lib/contract/run-records-core.ts:999-1005` uses copy+append to temp file and rename without inter-process locking.
  - `src/lib/runtime-trace/runtime-card-trace.ts:209-217` performs multiple append calls per invocation; another process can interleave writes on same runId.
- Failure scenario:
  1. Two runtime-card processes are started with identical `--trace-out` runId.
  2. Both read the same current events tail, append independently to separate temp files.
  3. Last rename wins; one process's event append can be dropped.
  4. Resulting event stream appears valid but is missing expected lifecycle events from one invocation.
- Impacted behavior:
  - Replay incompleteness under concurrent mutation.
- Remediation:
  - Enforce exclusive run directory ownership per invocation (lock file + fail if held), or reject existing runId upfront (finding #1 fix also mitigates this).
  - Add concurrency stress test around append path if shared-runId mode is ever permitted.

## Non-findings / Confirmed constraints

- Docs consistently frame trace output as advisory/audit/orientation only and not delivery/merge authority (`docs/cli-reference.md`, `docs/agents/00-architecture-bootstrap.md`, `docs/agents/07b-agent-governance.md`).
